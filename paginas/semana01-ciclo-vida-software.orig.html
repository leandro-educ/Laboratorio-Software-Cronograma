02

De Code & Fix a la ingeniería de software

El desarrollo improvisado puede funcionar en proyectos pequeños, pero genera problemas importantes cuando el sistema crece o intervienen múltiples actores.

Modelo tradicional · 1960

Code & Fix

  • Se programa lo que se necesita en el momento.
  • Si algo falla, se corrige sobre la marcha.
  • No hay supervisión formal ni etapas claras.
  • Escala mal cuando crece el sistema o el equipo.

Problema central

La crisis del software

Costos crecientes
Retrasos frecuentes
Baja calidad y errores difíciles de rastrear
Dificultad para mantener y evolucionar productos

Respuesta disciplinar

La ingeniería de software surge como una forma de desarrollar y mantener software de calidad mediante métodos, procesos y técnicas específicas.

Cambio de enfoque

El software deja de pensarse solo como código y comienza a comprenderse como proyecto, producto y proceso.

03

¿Qué es el ciclo de vida del software?

Es el marco de referencia que organiza las actividades y tareas involucradas en el desarrollo, la explotación y el mantenimiento de un producto software.

Norma ISO 12207

Definición operativa

El ciclo de vida del software contiene las actividades y tareas involucradas en el desarrollo, la explotación y el mantenimiento de un producto software, abarcando desde su definición hasta la finalización de su uso.

Importancia

  • Organiza el trabajo del equipo.
  • Asegura calidad y trazabilidad.
  • Reduce riesgos y costos.
  • Mejora la gestión del tiempo y los recursos.

Etapas del ciclo de vida

Cada etapa aborda decisiones distintas y produce información que impacta en las siguientes.

R

Requerimientos

Identificar qué necesita el cliente y qué problema se desea resolver.

A

Análisis

Traducir necesidades a lógica de negocio, alcance y riesgos.

D

Diseño

Definir arquitectura, tecnologías, interfaz y estructura técnica.

</>

Desarrollo

Codificar con estándares, documentación y control de versiones.

✓

Pruebas

Verificar calidad antes de la liberación: pruebas unitarias, de integración y de usuario.

∞

Implementación y mantenimiento

Poner en producción, monitorear, corregir y evolucionar el producto.

Secuencia de trabajo

Las etapas permiten ordenar decisiones, tareas y entregables dentro de un proyecto.

Diferenciación importante

El análisis se concentra en comprender el problema; el diseño se enfoca en construir la solución técnica.

Relación con la materia

Los temas de programación, testing, patrones, APIs y despliegue pueden ubicarse dentro de este proceso.

05

De requerimientos a mantenimiento

Este desglose muestra qué problema resuelve cada fase y qué tipo de decisiones se toman en cada una.

Requerimientos y análisis

En esta fase se define qué necesita el cliente, qué problema se quiere resolver y qué debe hacer el sistema.

Se define

Objetivo del software, requisitos funcionales y no funcionales, interfaz esperada, plazos, presupuesto y mantenimiento acordado.

Se analiza

Lógica de negocio, riesgos, restricciones, dependencias y casos de uso.

Entregables típicos

Listado de requisitos, alcance, restricciones y primeras decisiones de negocio.

Diseño

En el diseño se decide cómo se desarrollará la solución: arquitectura, tecnologías, base de datos, interfaz, integración y estructura de clases.

Se define

Arquitectura tecnológica, lenguajes, frameworks, base de datos y componentes.

Se modela

Diagramas, flujos, interfaces, integración con APIs, seguridad y escalabilidad.

Resultado

Un plan técnico claro para pasar a la implementación.

Desarrollo

En esta etapa se implementa el software siguiendo estándares de calidad, buenas prácticas, documentación y control de versiones.

Pruebas

Unitarias, de integración y de usuario para detectar errores antes del lanzamiento.

Mantenimiento

Correcciones, mejoras, nuevas funciones, backups y soporte técnico.

06

Modelos de ciclo de vida del software

Los modelos cambian la forma de recorrer las etapas, de gestionar el riesgo y de responder a los cambios.

Modelo en cascada

Lineal y estructurado

Útil cuando los requisitos están bien definidos y cambian poco.

Modelo V

Diseño y prueba asociada

Relaciona cada etapa de construcción con una etapa de validación.

Modelo espiral

Iterativo y con foco en riesgo

Permite planificar, evaluar riesgos, desarrollar y volver a mejorar.

Modelo ágil

Iteraciones cortas

Entrega continua, trabajo colaborativo y adaptación al cambio.

Representaciones visuales

Puntos para comparar

  • No existe un único modelo adecuado para todos los proyectos.
  • La elección depende del alcance, el riesgo, la claridad de requisitos y la necesidad de adaptación.
  • La diferencia entre modelos está en la sucesión de etapas y en cómo gestionan cambios y riesgo.
07

Cascada, V, incremental y espiral

Comparar modelos permite comprender que un mismo proceso general puede recorrerse de maneras diferentes.

Modelo en cascada

Avance por pasos sucesivos

Se imagina como una secuencia en la que cada paso se completa antes de pasar al siguiente.

  • Fuerte orden secuencial.
  • Mayor rigidez frente a cambios.
  • Útil cuando el problema está bien definido.

Modelo V

Construcción y validación acopladas

Relaciona requerimientos, diseño y codificación con pruebas unitarias, de integración, de sistema y de aceptación.

Modelo incremental

Agregar partes nuevas

Se construye por incrementos: cada entrega suma funcionalidad al producto.

Modelo espiral

Iterar con foco en riesgo

Se vuelve sobre el producto mejorando y refinando a medida que se evalúan riesgos.

  • Determinar objetivos.
  • Evaluar riesgos.
  • Desarrollar y probar.
  • Planificar la siguiente vuelta.
08

Modelo ágil y Scrum

El enfoque ágil propone construir en partes pequeñas, validar rápido, incorporar retroalimentación y ajustar continuamente.

Modelo ágil

Entrega iterativa e incremental

Es un enfoque de gestión y desarrollo basado en iteraciones cortas, colaboración continua y adaptabilidad al cambio.

Scrum

Organización rápida y flexible del trabajo

  • Sprints cortos.
  • Planificación, ejecución, revisión y retrospectiva.
  • Trabajo visible y coordinación del equipo.

Comparación rápida

Característica Incremental Espiral Ágil
Cómo se trabaja Se agregan partes nuevas completas. Se mejora y perfecciona lo mismo varias veces. Se hacen partes pequeñas rápido y con retroalimentación continua.
Cambio de planes Más difícil sobre lo ya hecho. Permite mejoras dentro de un marco de planificación. Admite cambios frecuentes sin romper la lógica de trabajo.
Imagen mental Agregar más partes nuevas. Mejorar lo que ya existe. Construir, probar, ajustar y seguir.
09

Síntesis final

El ciclo de vida del software permite comprender el desarrollo como un proceso organizado, justificable y sostenible.

Ideas principales

  • El software profesional no se improvisa.
  • El ciclo de vida organiza tareas, decisiones y entregables.
  • Los modelos cambian la manera de recorrer las etapas, no la necesidad de pensar el proceso.
  • Los contenidos de la materia pueden ubicarse dentro de estas fases.

Proyección hacia próximas clases

  • Requerimientos → modelado de dominio.
  • Diseño → arquitectura, patrones y decisiones tecnológicas.
  • Desarrollo y pruebas → Java, testing, persistencia y calidad.
  • Implementación → APIs, servicios y mantenimiento evolutivo.

Construir software no es avanzar a ciegas: es tomar decisiones ordenadas, justificadas y sostenibles.
